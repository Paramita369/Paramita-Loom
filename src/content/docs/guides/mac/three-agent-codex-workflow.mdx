## TL;DR

3-Agent workflow 的關鍵唔係 agent 越多越好，而係每個 agent 都只碰自己負責的 repo 與 scope，並由 Lead 明確決定 merge order 同 blocker escalation。

## Why it matters

呢頁提供單機 multi-agent 協作嘅最小秩序，避免 worktree 串染、repo 邊界鬆動，或者 Agent 2 倒推改 private contract。

## Core model / main idea

- Lead 只做 scope control、風險管理、merge sequencing，同 blocker escalation，唔直接偷做 implementation。
- Agent 1 要保護 frozen contract，同時準備 Agent 2 可消費的 canonical bundle 與 reviewed body。
- Agent 2 只能消費已批准的 private output，唔可以反向改寫 private contract 或 schema。
- worktree 只係隔離執行空間，真正邊界仍然係 repo、contract 同 acceptance gate。

## Examples / failure modes / usage

- 使用：Lead 先凍 scope、merge order 同 blocker condition，Agent 1 只做 private canonical output，Agent 2 只消費 approved output。
- 失效模式：dirty worktree 被 auto-apply 會把 unrelated local diff 帶入新 worktree，造成 branch 污染。
- 使用：每個 agent 都應由乾淨 commit 開 branch，提交前只 stage 自己責任範圍。
- 失效模式：如果 Agent 2 倒推修改 private contract，就會打破 frozen boundary 同 merge sequencing。
