## TL;DR

Phase 15 真正證明的唔係單一網站上線，而係 private canonical bundle、review queue、publish queue、preview verification 同 publish receipt 已經可以形成可回放的閉環。

## Why it matters

呢頁幫人睇清楚 Phase 15 已經證明咗邊條閉環存在，亦清楚分隔已落地能力同後續 public wiring 波次。

## Core model / main idea

- Phase 15 先凍結 contract，之後再補 private publish hooks，避免 public repo 提前變成真相來源。
- canonical bundle、review queue 同 publish queue 分別承擔內容定稿、審核流同待發佈流轉責任。
- preview verification 同 publish receipt 令 public 狀態變更可以被驗證、回放同追責。
- review_cycles 將之後的內容回顧節奏帶返 control DB，而唔係散落在 thread 或人工記憶。

## Examples / failure modes / usage

- 使用：沿住 WP0 contract freeze -> private publish hooks -> preview verification -> receipt 呢條主線理解 Phase 15。
- 失效模式：preview 失敗時如果仍然標示為 published，就會破壞 receipt contract 同 public status 信任。
- 使用：先睇 contract note，再睇 private publish contract，最後對照 build log 會最容易分辨已落地與 defer 範圍。
- 失效模式：如果將 public-side wiring 同 contract change 混入同一波，repo boundary 會變得唔清楚。
