## 先看結論

呢篇用來確認 OpenClaw 已經正常啟動，而唔係只係「某啲 process 仲在場」。你要同時看到 healthz、readyz、LaunchAgents、receipt 同 DoD checkpoint 都對齊，先可以放心。

## 為什麼值得關心

啟動檢查如果做得鬆散，最容易把文件整理、process 存活、或者 thread 訊息，誤認成真正的系統真相。呢頁只教你用 repo-backed evidence 收口。

<GuideMetadata />

## 核心模型

你可以照以下順序做：

1. 先跑 `python -m bootstrap.platform_ops preflight`，確認環境、路徑同最基本前置條件都合理。
2. 如果資料庫未落地，先做 `init-db`，再做 `render-launchd` 同 `verify-launchd`，避免 launchd 模板同實際路徑對唔上。
3. 載入並 kickstart 三個 LaunchAgents，之後再檢查 `runtime/logs/` 同服務狀態。
4. 用 `GET /healthz` 同 `GET /readyz` 去分清「活著」同「可以服務」。
5. 回到 Slack root task，核對 `@OpenClaw`、`task_contract`、receipt 同 review gate，確保根任務真係落到 DB truth。
6. 最後對照 DoD checkpoints，確認安裝與啟動唔止完成到表面，而係有證據可回放。

## 例子與常見誤解

- 使用：process 在場只代表啟動過，不代表 ready。
- 失效模式：未替換 `__OPENCLAW_HOME__` 就起 launchd，會令 health check 指向錯位路徑。
- 失效模式：DB 未初始化就跑 smoke，通常會得到唔穩定或假性的結果。
- 失效模式：把 Slack thread 當 truth，會令 root task、receipt 同 review gate 失去唯一來源。
- 失效模式：不要把文件整理誤寫成已完成實機驗證；這頁只屬「文件已核對階段」，不是現場跑過的證明。
