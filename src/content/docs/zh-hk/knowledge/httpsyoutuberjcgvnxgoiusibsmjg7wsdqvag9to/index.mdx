# https://youtu.be/rJCgvnXgOiU?si=BsmJG7WSDQVAg9TO

## 一句話版本

呢段內容講緊一個叫「RAG Anything」嘅做法，重點係補足一般 RAG 只識食文字文件嘅弱點，令掃描 PDF、圖片、圖表呢類真實世界文件都可以入同一套知識檢索流程。

## 點解重要

- **RAG 嘅實際限制唔係理論問題，而係文件型態問題。** 如果我哋手上好多資料其實係掃描件、截圖、圖表報告，單做 text-only RAG 會令大量營運知識根本入唔到系統，咁檢索再準都冇用。
- **「文件可用性」直接影響系統覆蓋率。** 片段反覆強調真實工作場景唔係純文字世界，意思係如果檢索系統食唔到多模態文件，最後只會變成對少量乾淨資料有用，對日常操作幫助有限。
- **RAG Anything 嘅價值唔只係加 parser，而係延續原本 LightRAG 嘅使用方式。** 佢唔係推翻原本架構，而係包住現有 LightRAG，再將非文字文件處理後匯入同一個 knowledge graph；對團隊嚟講，呢種擴充比另起一套系統更易落地。
- **統一查詢入口好重要。** 片段提到非文字文件最後會同原本文字資料 merge 去同一個知識圖譜，咁代表使用者唔使記住「邊類資料喺邊套庫」，減少操作心智負擔。
- **多模態支援唔係免費午餐。** 代價包括本機要下載額外模型、整體系統會重身啲；即係話功能提升同時，部署複雜度、磁碟空間、資源使用都會增加，唔可以當成零成本升級。
- **匯入流程會由 UI 轉向腳本化。** 片段講到非文字文件唔能夠直接靠原本 LightRAG UI ingestion，而係要用 script。呢點好關鍵，因為佢代表操作流程、權限控制、錯誤處理都要重新設計。
- **對終端使用者而言，複雜度可以被包裝起來。** 雖然底層多咗模型與腳本，但片段提出可以透過一層 agent / skill 呼叫去做 ingestion，令操作者只需講一句指令。真正有價值嘅地方係把技術成本集中喺平台層，而唔係轉嫁俾每個使用者。
- **作者刻意強調理解架構，而唔係只識安裝。** 呢個訊號值得留意，因為多模態 RAG 一旦出問題，通常唔係 prompt tweak 可以救到，而係 ingestion、圖譜合併、向量化流程其中一層出咗偏差。
- **系統核心仍然係知識圖譜合併，而唔係單純 OCR。** 從片段描述睇，RAG Anything 唔只係將圖片變文字，而係會先建立自己嘅知識圖譜同向量資料，再同 LightRAG 合流；即係檢索品質取決於融合策略，而唔止抽字準唔準。
- **佢適合做增量升級，不適合當萬能解藥。** 片段一方面講「容易接入」，另一方面又承認更重、更要腳本，反映出呢方案最適合已經有 LightRAG 基礎、而家要補 document coverage 嘅團隊，而唔係完全冇基礎設施嘅情況。

## 我哋點睇

- 如果我哋要做文件導向嘅知識系統，應該一開始就假設「大量來源唔係乾淨文字」，否則後面一定要補 migration 成本。
- 架構上比較合理嘅做法係保留單一查詢面，將多模態 ingestion 當作後台能力，而唔好畀使用者分辨「文字文件用 A、掃描 PDF 用 B」。
- 要落地就唔可以只驗 demo，需要額外驗三樣嘢：掃描 PDF 抽取品質、圖表/圖片內容入庫後嘅可查詢性、以及合併到同一 knowledge graph 之後會唔會令檢索結果變亂。
- 對 OpenClaw 呢類偏控制面、證據鏈導向嘅系統，最值得借鏡嘅唔係「多模態」本身，而係將較重嘅文件理解流程包成受控後台步驟，前台仍然維持簡單、可審計嘅操作入口。
