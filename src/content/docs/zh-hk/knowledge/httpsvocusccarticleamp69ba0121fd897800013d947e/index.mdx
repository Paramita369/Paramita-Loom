# https://vocus.cc/article/amp/69ba0121fd897800013d947e

## 一句話版本

作者分享佢點樣用本地運行、分工清晰、共享知識庫嘅 5 人 Agent 幕僚團隊去處理日常工作，重點唔係技術花巧，而係點樣將組織流程同知識管理設計成一套真係跑得順嘅系統。

## 點解重要

- **多 Agent 真正有價值嘅地方唔係「多開幾個 AI」**
- 作者最強調嘅其實係分工邏輯：邊個 Agent 見到咩訊號要接手、幾時要升級畀人決策、邊個輸出會成為下一個 Agent 嘅輸入。呢個對我哋有提醒，因為如果只係堆模型同 prompt，最後通常只會變成幾個各自講嘢、但唔會協作嘅機械人。
- **共享通用知識庫加專屬知識庫，先可以同時做到一致性同專業深度**
- 通用層確保所有 Agent 對公司方向有共同理解，專屬層先令法務、產品、技術等角色唔會答到太泛。對我哋嚟講，呢個比「一個大而全 RAG」更實際，因為唔同角色需要嘅上下文本身就唔一樣。
- **知識要先被整理、萃取、更新，Agent 輸出先會穩定**
- 文中講得幾白：幕僚讀嘅係整理過嘅摘要，不是原始資料堆。意思係，資料工程先係核心工序，唔係尾段配件。對團隊而言，投入時間喺資料粒度、更新頻率、摘要品質，回報會遠大過不停調 prompt。
- **Agent 可以幫手預判、拆任務、開票，但最後判斷權仍然要清楚留畀人**
- 例如 Email 入嚟後，Agent 會同時出回覆草稿、建立待辦、開決策票，再視情況叫法務進場。呢種設計重要在於佢唔係取代人，而係將人由低價值分發工作抽離，只保留最需要人拍板嘅位。
- **自動串聯比手動 dispatch 更接近真實組織運作**
- 作者唔係每次自己分派，而係預先定義「遇到呢種情況應該諮詢邊個」。呢點值得注意，因為一個可持續嘅 Agent 系統，應該靠規則同責任邊界運作，而唔係靠使用者每次親自 orchestrate。
- **高品質輸出其實高度依賴過往決策紀錄同內部文件成熟度**
- 作者話因為自己有決策紀錄庫、固定整理思路，所以 Agent 輸出先能夠貼近預期；產品幕僚亦因為食到完整產品知識庫，先做到接近可直接策略討論嘅水平。對我哋而言，呢個即係如果歷史決策無被結構化保存，Agent 上限會好快撞牆。
- **「Context」唔只係提示字數，而係組織記憶有冇被制度化**
- 文中所謂 Context 重要，唔係單純 token 多寡，而係有冇清楚記錄背景、原因、目標、標準。呢個觀點對實作好重要，因為我哋要設計嘅唔只係 prompt pipeline，而係一套可追蹤、可回溯、可持續修正嘅知識系統。
- **技術門檻會持續下降，所以真正差異化會轉去 workflow design**
- 作者直講技術本身之後只會愈來愈簡單，代表競爭點唔喺「會唔會串 API」，而喺「有冇能力把業務邏輯講清楚」。對團隊來講，越早把流程顯性化，越容易喺工具成熟時食到最大紅利。
- **角色人設同前台視覺雖然唔必要，但可能提升長期使用意願**
- 作者承認有啲設計純粹令自己開心，但亦反映一個現實：如果系統令人想持續打開、持續觀察，就更容易養成使用習慣。對內部工具來講，可用性唔一定只來自效率，有時仲來自心理上「願意返去用」。
- **個人生產力提升唔一定會令工作變少，反而可能放大吞吐量同責任範圍**
- 作者測試一週後效率提升，但結果係自己更忙。呢點幾關鍵，因為導入 Agent 之後，團隊可能唔係單純節省工時，而係會自然承接更多分析、跟進、決策工作，管理預期要提早做。
- **由個人使用升級到團隊使用時，架構同挑戰會變**
- 文末提到後續改用 API、進入團隊協作階段。對我哋有啟示：個人 Agent 成功唔代表團隊版會自然成立，因為共享權限、責任界線、知識維護、決策一致性都會即刻變成新問題。

## 我哋點睇

- 如果我哋想做自己嘅 Agent 系統，第一步唔應該係設計幾多角色，而係先盤點三樣嘢：決策紀錄有冇、任務流點樣轉手、邊啲情況一定要人批。呢三樣未清，Agent 只會幫我哋更快地亂。
- 對 OpenClaw 呢類控制系統嚟講，文中最有用嘅唔係「5 個幕僚」個形式，而係「共享真相來源 + 明確狀態轉移 + 誰有權做最後決定」呢套思路。呢個同我哋強調控制面、任務真相、審核閘口其實係同一路數。
- 值得吸收嘅實作方向係：將知識庫建設當成正式工程項目，而唔係部署後補做。因為一旦知識來源、摘要策略、更新規則無被制度化，後面加再多 Agent 都只係放大噪音。
- 如果之後要畀操作員睇進度，作者嗰種「幕僚指揮中心」思路可以借鏡，但用途應該偏向可觀測性而唔係娛樂感。即係畫面要幫人一眼睇到邊個 task 卡住、邊個需要批准、邊個已經進 review，而唔係淨係見到 agent 在跑。
