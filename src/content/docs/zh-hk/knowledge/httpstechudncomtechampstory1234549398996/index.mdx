# https://tech.udn.com/tech/amp/story/123454/9398996

## 一句話版本

Anthropic 推出嘅 `Claude Dispatch` 同開發向 `Channels`，代表「用手機/通訊工具遠端派 AI 去電腦做事」開始走向主流，但而家最大分野唔係功能多寡，而係安全、可控性同可用場景是否收斂得夠實際。

## 點解重要

- `Dispatch` 同 OpenClaw 做緊表面上相似嘅事，但產品哲學相反。前者主打封閉、審批、固定月費；後者主打自由、可擴充、自己承擔風險。呢個差異直接影響用戶會因為「安心感」而唔係「最強能力」去揀工具。
- 文章最有力嘅訊號唔係新功能本身，而係 Anthropic 正式證明「遠端發任務畀本機 agent」有足夠需求，已經值得做成獨立產品線，而唔再只係極客玩具。
- OpenClaw 被拿嚟對比時，作者首先講嘅係漏洞、暴露實例、惡意技能比例，而唔係自動化能力。即係話市場對呢類系統嘅第一印象，已經由「勁唔勁」轉成「安唔安全」。
- 社群技能市場被視為高風險來源，因為擴充性雖然帶來速度，但同時放大供應鏈問題。對我哋嚟講，任何「可安裝技能」設計如果冇清楚信任邊界，會被外界直接理解成風險入口。
- 固定月費被描寫成明顯優勢，原因唔只係平，而係可預期。對一般用戶而言，「月底唔知會燒幾多 token」本身就足以阻止採用，即使功能更強都未必贏。
- 上手門檻被放到同安全、成本並列，代表配置複雜度唔係次要 UX 問題，而係產品能否跨過工程師圈層嘅核心條件。API key、模型選擇、環境變數呢類概念，對大眾產品都係實質阻力。
- `Dispatch` 現階段最適合嘅唔係「代你操作整部電腦」，而係「幫你搵、整、彙總資料」。呢個界線好重要，因為佢反映咗現時穩定可交付嘅任務類型其實好窄。
- 文章列出成功率大約得一半，說明真實世界體驗仍然好早期。市場可能接受研究預覽，但如果外界期待被拉高到「可靠助手」，失敗會直接變成信任流失。
- 失敗任務集中喺操作即時狀態同活躍應用，例如 iMessage、Terminal session、Safari 分頁。即係需要讀取動態前景上下文嘅工作，穩定性明顯比靜態檔案/知識整理差。
- 運作限制幾具體：電腦要醒、桌面 app 要開、只有單一對話、冇完成通知、速度慢。呢啲都提醒我哋，遠端 agent 嘅價值未必輸喺模型，而係輸喺調度、狀態同步同回報機制。
- `Channels` 比 `Dispatch` 更值得開發團隊注意，因為佢將外部訊號直接送入本地開發工作階段，等 CI、監控、聊天室事件可以驅動 agent。呢個方向更加接近持續運作嘅工程自動化，而唔只係個人助理。
- 文中提到 Anthropic 近期把 `Remote Control`、`Scheduled Tasks`、`Channels` 拼埋一齊，顯示佢哋唔係零碎加功能，而係逐步搭建「人唔喺度都可持續工作」嘅 agent 形態。競爭已經由單次問答轉去 workflow orchestration。
- OpenClaw 被形容為「越獄 Android」，Dispatch 被形容為「封閉 iPhone」，呢個比喻其實講緊一個現實：大眾市場往往先接受受限但低風險嘅系統，之後先考慮進階自由度。
- 對開發者生態而言，MCP 插件化 connector 被點名係關鍵。即係外部平台整合唔一定要官方全包，但前提係核心邊界、安全模型同授權流程要先設計好，否則只會放大風險。
- 最後個趨勢判斷值得記住：AI 介面正由「坐喺電腦前提問」轉成「任何地方發任務，背景執行後交結果」。呢個轉變會令可觀測性、審批、恢復能力同任務真相來源變得比 chat UX 更重要。

## 我哋點睇

- 對 OpenClaw 而言，最應該強化嘅賣點唔係「可以做更多」，而係「點樣安全地只做應該做嘅事」。如果外界首先聯想到漏洞同惡意技能，再強嘅能力都會被蓋過。
- 我哋應該把「資料整理型任務」視為第一批最易建立信任嘅場景，例如檔案分類、摘要、收據整理、研究彙整，避免一開始把主訊息放喺高風險主動操作。
- 產品敘事要明確講清楚邊個動作需要批准、邊個元件有權改公開狀態、邊度先係任務真相來源。呢啲唔只係架構內部原則，亦係對外建立可信度嘅語言。
- 技能生態如果要存在，就要預設成受控分層，而唔係默認自由安裝。否則市場會直接用供應鏈風險框架去理解我哋。
- 固定成本感知同低配置門檻要被當成一等問題處理，即使用底層技術上未必真係固定月費，亦要盡量令操作者知道「大概會花幾多、開箱要做幾多步」。
- 從文章反映嘅趨勢睇，我哋做單機、資料庫中心、審批與驗證分離，其實方向係啱；真正要贏係把呢套控制力變成用家睇得明、感受得到嘅體驗，而唔只係架構描述。
